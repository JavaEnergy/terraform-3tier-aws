📖 ინფრასტრუქტურის განმარტება

2-რეგიონიანი, 3-დონიანი AWS ინფრასტრუქტურა · Terraform Portfolio Project

🎯 პროექტის მიზანი

ეს პროექტი არის Terraform-ის პორტფოლიო პროექტი, რომელიც ავტომატურად ქმნის სრულ, პროდაქშენ-მზა AWS ინფრასტრუქტურას. Terraform-ის კოდი ერთ ბრძანებაში (terraform apply) ქმნის დაახლოებით 80 AWS რესურს ორ სხვადასხვა გეოგრაფიულ რეგიონში — us-east-1 (ვირჯინია) და us-west-2 (ორეგონი). ინფრასტრუქტურა დაყოფილია 3 ძირითად დონედ (Tier): Frontend, Backend და Database.

🌐 Tier 1 — Frontend (S3 სტატიური ვებ-ჰოსტინგი)

🪣 რა არის S3?

Amazon S3 (Simple Storage Service) არის AWS-ის ობიექტური საცავი. ჩვენ ვიყენებთ მას სტატიური ვებ-გვერდის (HTML, CSS, JS) სასტუმროდ. მომხმარებელი ბრაუზერიდან პირდაპირ სთხოვს S3-ს ფაილებს — არ გვჭირდება ვებ-სერვერი.

⟳ Cross-Region Replication (CRR)

us-east-1-ში ატვირთული ყველა ფაილი ავტომატურად კოპირდება us-west-2-ში. ეს უზრუნველყოფს მაღალ ხელმისაწვდომობას (High Availability) — თუ ერთი რეგიონი გაითიშება, მეორე კვლავ მუშაობს. CRR-ისთვის საჭიროა Versioning ჩართვა ორივე bucket-ში და IAM Replication Role.

💡 Terraform-ში: S3 module ერთხელ იძახება ორი provider-ით (aws.primary + aws.secondary) — ეს არის Terraform-ის advanced pattern dual-provider module-ისთვის.

⚙️ Tier 2 — Backend (ECS Fargate + ALB)

🌐 VPC — ვირტუალური ქსელი

Virtual Private Cloud — ეს არის ჩვენი კერძო ქსელი AWS-ში. us-east-1-ში: 10.0.0.0/16, us-west-2-ში: 10.1.0.0/16. ინტერნეტიდან პირდაპირ VPC-ში შესვლა შეუძლებელია — ყველა ტრაფიკი კონტროლდება.

⚖️ ALB — Load Balancer

Application Load Balancer — Public Subnet-ში ზის და ინტერნეტიდან მოსულ HTTP :80 მოთხოვნებს გადაამისამართებს Private Subnet-ში მყოფ ECS კონტეინერებზე (:5000). ეს არის ინფრასტრუქტურის "კარი".

🔀 NAT Gateway

Private Subnet-ში მყოფ ECS კონტეინერებს ინტერნეტი სჭირდებათ (Docker image-ის გადმოსაწერად, Secrets Manager-ზე წვდომისთვის). NAT Gateway ამ outbound ტრაფიკს ამუშავებს, მაგრამ inbound-ს ბლოკავს.

🐳 ECS Fargate — Serverless კონტეინერები

Elastic Container Service (ECS) Fargate-ით ნიშნავს, რომ სერვერები არ გვჭირდება — AWS თავად მართავს infrastructure-ს. ჩვენ მხოლოდ Docker image-ს ვაწვდით (ECR-ში) და ვეუბნებით: 256 CPU unit, 512MB RAM, desired_count=1. Flask Python-ის აპლიკაცია მუშაობს port 5000-ზე.

🔐 Secrets Manager + ECR

ECR (Elastic Container Registry) — AWS-ის Docker registry, სადაც ვინახავთ ჩვენს image-ებს.
Secrets Manager — DB_PASSWORD-ს ინახავს დაშიფრულად. ECS task-ი runtime-ზე თავად კითხულობს პაროლს — კოდში ან terraform.tfvars-ში არ ჩანს.

💡 Security Groups: ALB-ს SG-ი იღებს :80 ნებისმიერი IP-დან. ECS-ს SG-ი იღებს :5000 მხოლოდ ALB-ს SG-იდან. RDS-ს SG-ი იღებს :5432 მხოლოდ ECS-ს Private Subnet CIDR-იდან. ეს არის Defense-in-Depth პრინციპი.

🗄️ Tier 3 — Database (RDS PostgreSQL)

🐘 რა არის RDS?

Relational Database Service — AWS-ის მართული მონაცემთა ბაზა. ჩვენ ვიყენებთ PostgreSQL 15.4 db.t3.micro instance-ზე (20GB gp2 დისკი). "მართული" ნიშნავს: backup, patching, failover — ყველაფერს AWS ანხორციელებს. ბაზა მხოლოდ Private Subnet-შია — ინტერნეტიდან პირდაპირი წვდომა შეუძლებელია.

🔑 პაროლის მართვა

random_password Terraform resource ქმნის 16-სიმბოლოიან random პაროლს. ეს პაროლი ინახება Secrets Manager-ში JSON ფორმატში: {"username":"appuser","password":"..."}. ECS task definition-ში secrets block კითხულობს DB_PASSWORD-ს runtime-ზე.

📊 მონიტორინგი: CloudWatch + SNS

CloudWatch — AWS-ის monitoring სისტემა. ჩვენ ვქმნით:
• Log Group — ECS კონტეინერის ყველა log-ი (print(), error-ები) ინახება CloudWatch-ში
• CPU Alarm — თუ ECS-ის CPU 80%-ს გადააჭარბა 2 წუთით, alarm ამოქმედდება

SNS (Simple Notification Service) — alarm-ი SNS Topic-ს აგზავნის შეტყობინებას, SNS-ი კი email-ს უგზავნის chogovadzebeka@gmail.com-ს.

⚠️ მნიშვნელოვანია: terraform apply-ის შემდეგ Gmail-ში მოვა confirmation email — უნდა დაადასტუროთ subscription, წინააღმდეგ შემთხვევაში alarm-ები არ მოვა.

📁 Terraform კოდის სტრუქტურა

კოდი დაყოფილია reusable modules-ად. თითოეული module ერთი პასუხისმგებლობის პრინციპს (Single Responsibility) იცავს. Root main.tf იძახებს თითოეულ module-ს ორჯერ — primary და secondary region-ისთვის.

module "vpc_primary" {
  source = "./modules/vpc"
  providers = { aws = aws.primary }
}

module "vpc_secondary" {
  source = "./modules/vpc"
  providers = { aws = aws.secondary }
}
Apply-ის თანმიმდევრობა (Dependency Graph)
VPC → NAT Gateway → ALB
VPC → Secrets → RDS
RDS + NAT + CloudWatch → ECS
SNS → CloudWatch Alarm
S3 (დამოუკიდებელი)
💡 Circular Dependency Fix

CloudWatch Alarm-ს ECS cluster/service სახელი სჭირდება, ECS-ს კი CloudWatch log group. ეს წრიული დამოკიდებულება გადაჭრილია locals block-ით root main.tf-ში — სახელები წინასწარ გამოითვლება.
